# Top 20 Key Learnings from Python MCQs

### Core Python Quirks
1. **The Mutable Default Argument Trap:** Defaults evaluate only once at function definition time. Never use `def func(a=[])`, instead use `a=None`.
2. **Integer Interning & Identity (`is` vs `==`):** Python caches small integers `[-5, 256]`. `256 is 256` evaluates to True, but `257 is 257` is False in the REPL because they are dynamically different objects.
3. **Tuple Immutability Limits:** A tuple is immutable, but the references it holds can point to mutable structures. Tuples can contain lists that can be freely mutated in-place.
4. **Scope Hoisting Nuance:** Assigning to a variable anywhere in a function immediately marks it strictly as a local variable for the entire scope, breaking outer variable lookups if done without `global` or `nonlocal`.
5. **Late Binding in Closures:** Functions (like lambdas in loops) bind to the outer variable reference, not the explicit value at calculation time.
6. **Dictionary Keys Must Be Hashable:** Mutable inputs like Lists and Dicts strictly lack `__hash__` and cannot map as Dictionary Keys.

7. **The `for...else` Mechanism:** The `else` clause attached to a `for` loop triggers strictly only if the loop naturally finishes without hitting a `break`.
8. **Finally Returns Supercede All:** A `return` mapped strictly inside a `finally` block structurally overrides and permanently replaces any prior `return` executed inside `try` or `except`.
9. **List Modification Loop Destructor:** Mutating a list `[1, 2, 3]` natively whilst iterating over it dynamically causes elements to visually shift backwards, making the iterator implicitly identically skip items entirely.
10. **`isinstance` vs `type`:** Always favor `isinstance`. It structurally identically natively correctly traverses inheritance checking for parent classes, while `type()` rigidly identically mathematically checks identical class types natively.
11. **Empty Iterables:** `any([])` is natively `False`. `all([])` is structurally `True`.
12. **Method Resolution Order (MRO):** C3 Linearization enforces rigid sequential child-before-parent tracking, throwing TypeErrors natively identically explicitly when structurally identical parents logically inherently implicitly conflict natively.
